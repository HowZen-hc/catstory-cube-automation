# 泛化辨識可行性研究（爆擊傷害批次修復）

> 本文為 [0-feasibility-study.md](./0-feasibility-study.md) 的子研究。
> 主研究探討整體 OCR 匹配策略（字串替換 vs 模糊比對 vs 字元級混淆映射）。
> 本研究聚焦一個具體提案：**用萬用字元 regex 取代逐個誤讀加 fix pair**。

## 1. 問題本質

### 1.1 表面需求

- 有 30 個新的 OCR 失敗案例需要分類、修復
- 其中一位開發者建議使用「泛化辨識」—— `爆○害` / `爆○○害` / `爆○○○害` / `爆○○○○害`（含 害 / 喜）—— 以「省樣本」方式一次涵蓋多種爆擊傷害變體

### 1.2 核心問題（5 Why）

1. **Why**：為什麼要新增 30 個 case 的修復？→ PaddleOCR 不斷產生新的字元誤讀 / 字元遺漏 / 字元插入組合
2. **Why**：為什麼每個新組合都需要新 fix？→ `_OCR_FIXES` 是「字串 pair」替換，無法自動涵蓋新變體
3. **Why**：為什麼不直接用萬用 regex？→ 現行 regex `爆.\s*傷害` 只允許「爆 + 1 char + 傷害」，遺漏「傷字被吃掉」的情形
4. **Why**：為什麼不放寬成 `爆.{1,4}[害喜]`？→ 子字串重疊的屬性（`傷害 / 爆擊傷害 / 最終傷害 / Boss 傷害 / 爆擊機率`）在萬用 regex 下會互相污染
5. **Why**：到底「泛化」該泛化到什麼程度？→ **關鍵問題：介於「純白名單替換」與「純萬用 regex」之間的光譜**

### 1.3 成功標準

| 準則 | 目標 |
|------|------|
| 現有 FAIL case | 全部修復（可修復者） |
| False positive 風險 | 零（誤判會浪費方塊，代價 >> 漏判） |
| 現有 200+ 測試 | 全數通過 |
| 維護成本 | 不可無限成長 |

## 4. 「泛化辨識」定義與光譜

開發者原始提案：

```
爆○害
爆○○害
爆○○○害
爆○○○○害
爆○喜
爆○○喜
爆○○○喜
爆○○○○喜
```

實質上是把 `_OCR_FIXES` pair 累積的工作外包給 regex 萬用字元。以下為可能實作光譜：

| 強度 | Regex | 語意 |
|------|-------|------|
| 嚴格（現狀） | `爆.\s*傷害` | 爆 + 1 char + 傷害 |
| 輕放寬 M2 | `爆.\s*傷?害` | 爆 + 1 char + (optional 傷) + 害 |
| 中放寬 M1 | `爆.{1,4}[害喜]` | 爆 + 1-4 任意字 + 害或喜（**原提案**） |
| 受限放寬 M1' | `爆[^時終率機傷]{1,3}\s*傷?[害喜]` | 中段排除 `時/終/率/機/傷`，尾段 (optional 傷) + 害/喜 |

> 「泛化辨識」不是單一技術，而是一個光譜。**問題不是「是否泛化」，是「放鬆到哪個剛好涵蓋 FAIL case 不越過 FP 風險邊界」**。

## 6. 可能方案

### Option M2（保守 pair patch，不動 regex）

**核心**：現有 regex 不變；加入 10 條 scoped pair 與 regex fix：

```python
# _OCR_FIXES 新增（順序：要在 爆華 → 爆擊 等字元級 pair 之前）
("爆華害", "爆擊傷害"),   # 擊→華 + 傷 dropped
("爆挛害", "爆擊傷害"),
("爆擎害", "爆擊傷害"),
("爆馨擊", "爆擊"),       # 多出字元 馨
# 傷害後綴
("悔害", "傷害"),
("侮害", "傷害"),
("售害", "傷害"),
# 英文屬性
("工T", "INT"),

# _OCR_DEX_FIXES regex 新增：DX
# _OCR_INT_FIXES regex 新增：JNT、TIT
```

| 維度 | 評級 | 說明 |
|------|:----:|------|
| 可行性 | 🟢 | 沿用現有架構 |
| 工作量 | 🟢 | < 1 day（~10 條 entry + 11 新測試） |
| FP 風險 | 🟡 | **低但非零**：`爆X害` 三條 pair（`爆華害/爆挛害/爆擎害`）有 `爆` 前綴錨定，風險低；但 `悔害/侮害/售害 → 傷害` 與 `工T → INT` 無 `爆` 錨，任何含子字串 `悔害` 的文字都會被改寫（例如若未來 OCR 產生 `X悔害` 將變成 `X傷害`），需人工審查是否會與其他屬性衝突 |
| 可擴展性 | 🔴 | 未來新變體仍需新 pair；`_OCR_FIXES` 會繼續成長 |
| 維護成本 | 🔴 | 線性成長；已達 40+ 條 |

**修復比例**：11/11 FAIL case

**順序注意**：
- `爆華害 → 爆擊傷害` 必須排在 `爆華 → 爆擊` 之前，否則會先被降階成 `爆擊害`（regex 命中不到）
- `悔害/侮害/售害 → 傷害` 等無 `爆` 錨的 pair，必須確認**不會影響**任何現有屬性（人工盤點：`悔 / 侮 / 售` 三字在遊戲屬性名中不存在，風險可接受）

### Option M1（原提案萬用字元 regex）

```python
"爆擊傷害%": re.compile(r"爆.{1,4}[害喜]\s*[:\uff1a]?\s*\+?\s*(\d+) ?%"),
```

| 維度 | 評級 | 說明 |
|------|:----:|------|
| 可行性 | 🟢 | 單行修改 |
| 工作量 | 🟢 | < 1 hour |
| FP 風險 | 🔴 | **明確 FP：`爆擊時傷害+40%` 被 Boss 傷害腐蝕產生時誤判；`爆擊機害` 若率→害則誤判** |
| 可擴展性 | 🟢 | 自動涵蓋未見過的字元組合 |
| 維護成本 | 🟢 | 幾乎為零 |

**修復比例**：7/11（僅涵蓋爆擊傷害的 A/B/C 組），英文屬性仍需 pair

**Codex 獨立驗證結論**：**不建議採用**。`.` 萬用字元無法排除子字串重疊風險。

### Option M1'（受限泛化 regex，本研究提出）

```python
"爆擊傷害%": re.compile(r"爆[^時終率機傷]{1,3}\s*傷?[害喜]\s*[:\uff1a]?\s*\+?\s*(\d+) ?%"),
```

**關鍵設計**：
- 中段排除 `時`（Boss 傷害）、`終`（最終傷害）、`率 / 機`（爆擊機率）、`傷`（避免 `爆集害 → 爆傷害` 被誤抓）
- 尾段 `傷?` 允許 `傷` 被吃掉
- `[害喜]` 吸收 `害→喜` 誤讀
- `{1,3}` 上界限制跨列 concat 的污染範圍

| 維度 | 評級 | 說明 |
|------|:----:|------|
| 可行性 | 🟢 | 單條 regex 修改 |
| 工作量 | 🟢 | ~2 hours（regex + 擴充 FP 測試） |
| FP 風險 | 🟡 | **Codex 指出殘留風險**：若未來出現新的 `爆X害` 變體（X 不在 `_OCR_FIXES`），且來自跨列合併，仍可能誤判 |
| 可擴展性 | 🟢 | 自動涵蓋中段字元變體 |
| 維護成本 | 🟢 | 顯著降低 |

**修復比例**：7/7 爆擊傷害 FAIL case + 4 英文屬性（需併入 M2 英文 patch）= **11/11**

**實測**（`uv run python` 直接驗證，**前提是先經 `_fix_ocr_text` 正規化**，即沿用現有兩階段流程 fix → regex）：
- 15/15 已知爆擊傷害 case PASS（含現已通過的 8 + 新失敗的 7）
- 8/8 FP candidate 被正確拒絕（含 Codex 的 `爆集害 / 爆擊時傷害 / 爆擊機害 / 跨列 concat`）
  - 特別是 `爆集害+9%`：`_fix_ocr_text` 先將 `集害 → 傷害` 得到 `爆傷害+9%`，M1' 因 `傷` 在中段排除集而拒絕匹配，正確回退至 `傷害%` pattern

**Codex 殘留顧慮**：未來如果出現 `爆Y害` 變體（Y 是新字元，且 `Y害` 不在 `_OCR_FIXES`），且剛好由跨列合併產生，仍可能誤判為爆擊傷害。

### Option M3（M1' + M2 patch 組合）

同時套用：
1. M1' regex 修改
2. M2 的英文屬性 patch（DX / JNT / TIT / 工T）
3. M2 的 `悔害 / 侮害 / 售害 → 傷害` pair（**同時作為 M1' 跨列 concat FP 的第二層防禦**）

為什麼保留 `悔害 → 傷害` 等 pair：
- M1' 會吸收這些變體，pair 看似多餘
- 但有了 pair，`售害+9%` 的單列 OCR 會先被正規化為 `傷害+9%`，直接命中 `傷害%` pattern
- `parse_potential_lines` 的跨列合併只在兩列皆為「未知」時觸發；`售害+9%` 先被 pair 拉回 `傷害`，不再是「未知」，就不會被合併成 `爆售害` 觸發 M1'
- 雙重防禦：pair 擋單列，M1' 擋多字變體

| 維度 | 評級 | 說明 |
|------|:----:|------|
| 可行性 | 🟢 | 組合既有手段 |
| 工作量 | 🟢 | 1 day（regex + 10 pair + ~25 新測試） |
| FP 風險 | 🟢 | Codex 的 concat case 被 pair 層消除 |
| 可擴展性 | 🟢 | regex 涵蓋中段 + pair 涵蓋長度異常 |
| 維護成本 | 🟢 | 中長期穩定 |

**修復比例**：11/11

## 7. Codex 討論記錄

> **重要區分**：以下 Codex 提出的 FP 案例（`爆擊時傷害`、`爆擊機害`、跨列合併）**都是理論對抗性推演，不是真實觀察到的 OCR 輸出**。遊戲內爆擊傷害只有 `爆擊傷害 +1%` 或 `爆擊傷害 +3%` 兩種形狀，`時` 字不會出現在爆擊傷害列；`率 → 害` 的字元誤讀在視覺上差異極大，亦未觀察到。
>
> 這些推演的價值在於**指引 regex 的防禦性設計**（例如排除字元集），而**不是作為方案比較的案例證據**。下方保留討論記錄作為設計決策的來源，但請讀者不要把它們當成已知案例。

### 7.1 第一輪：原提案 M1 理論風險分析

**Codex 關鍵觀察**（獨立研究，未餵結論）：
- `爆.{1,4}[害喜]` 在理論上可被 `爆擊時傷害+40%`（假設 Boss 傷害腐蝕成此形狀）命中
- 理論上亦可被 `爆擊機害+25%`（假設率→害誤讀）命中
- 理論上跨列合併 `爆擊時` + `害+40%` 可組成 `爆擊時害+40%` 觸發 M1

**設計啟示**：不論這些假設是否會實際發生，regex 設計應把這些**字元位置作為排除集**（低成本的保險）。因此導出 M1' 的 `[^時終率機傷]` 字元類別。

### 7.2 第二輪：M1' 的殘留風險討論

**Codex 觀察**：
- M1' 在程式層消除了 `爆集害 → 爆傷害` 的路徑（因 `傷` 被加入排除集）
- 但仍無法保證未來**未見過**的 `爆Y害` 變體（Y 不在 `_OCR_FIXES` 且不在排除集）不會在跨列合併下誤判
- Kill criterion：一旦出現第一次**實際觀察到**的 damage-family FP，立即升級到 Option E

**設計啟示**：殘留風險是「未知未來變體」的理論擔憂，不是已知問題。以「持續監控 + kill criterion」處理，而非預先過度設計。

### 7.3 Claude 與 Codex 的分歧與整合

| 觀點 | Claude 初步 | Codex 獨立 | 整合結論 |
|------|------------|-----------|---------|
| 原 M1 是否可行 | 可能可行 | 理論 FP surface 過大 | **M1 不可取，應收斂成 M1'** |
| M1' 是否足夠 | 實測通過所有已知案例 | 理論殘留風險仍在 | **M1' + pair 層雙重防禦（= M3）** |
| M2 vs M1' 首選 | 偏向 M1'（涵蓋更廣） | 偏向 M2（不開 wildcard 表面） | **不互斥，M3 組合兩者** |
| 長期方向 | Option E | Option E | **一致** |

### 7.4 為什麼最終選 M3 組合

**Claude 的理由**：M1' 單獨可修 11/11 case，但跨列合併的理論路徑沒法用 regex 本身關閉。

**Codex 的理由**：M2 pair 層在單列已把 `售害 / 悔害 / 侮害` 正規化為 `傷害`，就不會產生「兩列都 `未知` → 觸發合併 → 命中 M1'」的路徑。

**整合（M3）**：pair 層擋單列、M1' 擋多字變體。雙方顧慮都消失。

## 9. 建議

### 9.1 主方案：**Option M3（M1' regex + M2 pair 組合）**

理由：
- 修復 11/11 可修復案例（A + B + C + D 組）
- Pair 層作為單列防禦，M1' 作為多字變體與組合泛化
- Codex 的跨列 concat FP 被 pair 層先消除（`售害+9%` 在單列已解析為 `傷害`，不進入合併路徑）
- 工作量 ~1 day，可擴展性良好
- 與現有架構高度相容，無破壞性變更

### 9.2 備選：**Option M2（純 pair patch）**

適用情境：想完全避免任何 regex wildcard 變動，接受線性 pair 成長。

代價：
- `_OCR_FIXES` 長期繼續膨脹
- 每個新變體都需手動加 pair
- 無法涵蓋「未見過的中段字元」

### 9.3 不推薦：**Option M1（原提案萬用 regex）**

Codex 已證明明確 FP：`爆擊時傷害` / `爆擊機害` / 跨列 concat。不採用。

### 9.4 長期方向：**Option E（字元級混淆映射）**

觸發條件（對 M3 的 kill criterion）：
1. 首次確認 damage-family FP（`爆擊傷害 / 傷害 / Boss 傷害 / 最終傷害 / 爆擊機率` 之間誤判）
2. `_OCR_FIXES` 超過 80 條
3. 連續兩個月每月新增 > 10 條 pair
4. 單一月內新增的 pair 中 > 25% 是既有混淆字家族的變體

達到任一條件時升級到 Option E。

## 10. 對「泛化辨識是否適合本專案」的直接回答

> **是否泛化、與是否模糊辨識的差別**

- **泛化辨識** = 放寬 pattern 讓一條 rule 涵蓋多種輸入變體；可以是 regex wildcard、可以是字元類別（排除特定字）、可以是 optional group
- **模糊辨識**（fuzzy matching）= 計算編輯距離 / 相似度；不是 pattern-based
- 兩者是不同技術。開發者的 `爆○害` 提案屬於 **regex 泛化**，不是模糊辨識

> **泛化是否適合本專案**

| 適用面 | 判斷 | 理由 |
|-------|------|------|
| 爆擊傷害（CJK 長屬性、有前後錨） | ✅ 有限度適用 | 用 M1' 受限放寬可行；務必排除 `時/終/率/機/傷` |
| 爆擊機率（與爆擊傷害共享前綴） | ❌ 不要泛化 | 易被爆擊傷害 pattern 搶去 |
| STR / DEX / INT / LUK（英文短屬性） | ❌ 不適合 | 字元太短；`_OCR_DEX_FIXES` 等已用字元類別 regex，已是最佳折衷 |
| 傷害 / Boss 傷害 / 最終傷害（子字串重疊） | ❌ 不要泛化 | 必須維持精準分隔 |

**結論**：泛化只在極有限的位置（CJK 長屬性 + 強邊界錨）適用；其餘屬性仍應使用 pair 或字元類別 regex。**「一招泛化治所有誤讀」的想法不適合本專案**，但局部泛化（M1'）配合 pair 保護（M2）可以省下未來大量維護成本。

## 12. 方法論附註（來自本次研究的 user 指正）

| 原則 | 說明 |
|------|------|
| 案例只收真實觀察 | 不能因為 Codex 或任何分析工具舉出對抗性 case，就把它寫進案例比較表。推測出來的 case 會失控且無法驗證是否真的發生 |
| 推論歸程式設計 | 對抗性推論的價值在指引**程式面防禦**（例如排除字元集），而不是作為方案比較的證據 |
| fix-all-issues 優先於 scope discipline | 研究過程中發現的既有 bug（例如 3.4 的 Boss regex）應一併修復，不以 scope 為由延後 |

這些原則應套用到後續所有 feasibility study 的寫法。
