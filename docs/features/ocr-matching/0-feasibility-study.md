# OCR 文字匹配策略可行性研究

## 1. 問題本質

### 1.1 表面需求

把 OCR 文字匹配從 pattern + 字串替換改成模糊比對，以提高辨識率。

### 1.2 核心問題（5 Why）

PaddleOCR 繁體中文模型會產生**字形相似的單字替換**（如 傷→佩/集/低/值），目前用手動維護的替換規則處理，每次遇到新誤讀都要加規則，維護成本無上限。

### 1.3 成功標準

- 辨識準確率 >= 現有方案
- 新誤讀不需逐案手動加規則（或大幅減少）
- 零 false positive（誤判會浪費方塊，代價極高）

## 2. 約束條件

| 類型 | 約束 | 彈性 |
|------|------|------|
| 技術 | 屬性是封閉集合（~30 種已知屬性） | 無 |
| 技術 | OCR 誤讀是字元級替換（非英文式拼錯） | 無 |
| 業務 | False positive 代價 >> false negative（誤判浪費方塊） | 無 |
| 技術 | 屬性名有子字串重疊（傷害 / 爆擊傷害 / 最終傷害 / Boss傷害）| 無 |
| 資源 | 個人專案，不想投入大量訓練工作 | 中 |

## 3. 現有能力盤點

### 3.1 相關模組

- `app/core/condition.py` — `_OCR_FIXES`（字串替換表）、`ATTRIBUTE_PATTERNS`（regex 匹配）、`_fix_ocr_text()`
- `app/core/ocr.py` — PaddleOCR 封裝（`lang="chinese_cht"`）、影像前處理

### 3.2 設計模式

- 兩階段處理：先 `_fix_ocr_text()` 修正已知誤讀 → 再用 regex 擷取屬性名 + 數值
- Regex 已包含部分字元容錯（如 `爆[擊擎]傷害`）

### 3.3 Tech Debt

- `_OCR_FIXES` 表已累積 40+ 條規則，跨 4+ 次 commit 逐步成長
- 每次新誤讀都需手動加規則 → 維護負擔無上限

## 4. 可能方案

### Option A：維持現狀（Pattern + 替換表）

**核心**：繼續手動新增 `_OCR_FIXES` 規則。

| 維度 | 評級 | 說明 |
|------|:----:|------|
| 準確度 | 🟢 | 已知誤讀 100% 正確 |
| False positive 風險 | 🟢 | 零，每條規則人工驗證 |
| 工作量 | 🟢 | 零（已完成） |
| 可擴展性 | 🔴 | 組合爆炸，N 個誤讀字 x M 個屬性 = N*M 條規則 |
| 維護成本 | 🔴 | 無上限，每個新誤讀都要加規則 |

### Option B：純模糊比對（difflib / fuzzywuzzy）

**核心**：用 SequenceMatcher 計算 OCR 文字與已知屬性名的相似度。

| 維度 | 評級 | 說明 |
|------|:----:|------|
| 準確度 | 🔴 | 中文字元級替換下，2 字屬性 1 字錯 = 50% mismatch |
| False positive 風險 | 🔴 | 極高。「傷害」vs「爆擊傷害」vs「最終傷害」會互相混淆 |
| 工作量 | 🟡 | 中等，但除錯成本高 |
| 可擴展性 | 🟡 | 不需手動加規則，但需調 threshold |
| 維護成本 | 🔴 | 需持續調整 threshold，難以穩定 |

**致命問題**：子字串重疊的屬性名在模糊比對下會互相匹配。目前 regex 用 negative lookbehind 精確區分，模糊比對做不到。

### Option C：混合方案（Regex 優先 + 模糊 Fallback）

**核心**：先用現有 regex 匹配，匹配不到再用模糊比對。

| 維度 | 評級 | 說明 |
|------|:----:|------|
| 準確度 | 🟡 | Regex 部分高，fallback 部分風險同 B |
| False positive 風險 | 🟡 | Fallback 處理的正好是最難的 case |
| 工作量 | 🟡 | 中等 |
| 可擴展性 | 🟡 | 改善有限 |
| 維護成本 | 🟡 | 仍需維護 regex + 調 fuzzy threshold |

### Option D：Fine-tune PaddleOCR 模型

**核心**：用遊戲字體截圖訓練專用 OCR 模型。

| 維度 | 評級 | 說明 |
|------|:----:|------|
| 準確度 | 🟢 | 理論最佳 |
| False positive 風險 | 🟢 | 從源頭解決 |
| 工作量 | 🔴 | 極高：收集標記資料、訓練、打包模型 |
| 可擴展性 | 🟢 | 模型自動泛化 |
| 維護成本 | 🟡 | 遊戲更新字體時需重新訓練 |

**不推薦**：30 個屬性 + 幾個誤讀字，用 fine-tune 是殺雞用牛刀。

### Option E：字元級混淆映射表（推薦）

**核心**：把「字串→字串」替換表重構為「字元→字元」混淆映射 + 已知屬性名精確匹配。

```python
# Layer 1: 字元級混淆映射
_CHAR_CONFUSIONS: dict[str, str] = {
    "佩": "傷", "集": "傷", "值": "傷", "低": "傷",
    "擎": "擊", "革": "擊", "事": "擊",
    "住": "性", "厨": "屬", "屈": "屬",
    # ...每個字一條，而非每個組合一條
}

# Layer 2: 已知屬性名（按長度降序，最長匹配優先）
_KNOWN_ATTRIBUTES = sorted([
    "爆擊傷害", "最終傷害", "物理攻擊力", "魔法攻擊力",
    "傷害", "全屬性", ...
], key=len, reverse=True)
```

| 維度 | 評級 | 說明 |
|------|:----:|------|
| 準確度 | 🟢 | 嚴格優於 A — 自動處理未見過的字元組合 |
| False positive 風險 | 🟢 | 只接受白名單替換 + 長度必須一致 |
| 工作量 | 🟢 | ~50 行新程式碼，主要是重組現有資料 |
| 可擴展性 | 🟢 | 新增混淆字 = 加 1 條 dict entry（不是 N 條） |
| 維護成本 | 🟢 | 大幅減少 |

**核心洞察**：問題不是字串相似度，而是字元級混淆映射。新增一個混淆字元只需 1 條 entry，而非 N 條字串 pair。

## 5. Codex 討論記錄

### 5.1 關鍵觀點

| 觀點 | Claude | Codex | 結論 |
|------|--------|-------|------|
| 模糊比對可行性 | 有子字串重疊風險 | 明確拒絕：中文字元級錯誤不適合 edit distance | 一致同意不適合 |
| Fine-tune 可行性 | 殺雞用牛刀 | 不成比例的工作量 | 一致同意不推薦 |
| 最佳方案 | 混合或改良替換表 | 提出 Option E（字元級混淆映射）| 採納 E |
| 核心洞察 | — | 問題不是字串相似度，是字元級混淆映射 | 關鍵轉折 |

### 5.2 Codex 識別的風險

- 部分誤讀是字元刪除/插入而非替換（如「爆機率」→「爆擊機率」），需保留少量字串級規則
- 屬性匹配需按長度降序（最長匹配優先）避免子字串問題

## 7. 建議

**推薦：Option E（字元級混淆映射）**

理由：
- 同時解決準確度和維護成本問題
- 零 false positive 風險（白名單替換 + 長度精確匹配）
- 改動量小（~50 行，重構現有資料結構）
- 新增混淆字從「加 N 條字串」降為「加 1 條字元」

**備選：Option A（維持現狀）**

如果短期內誤讀種類已趨於穩定，現有方案也可接受。

**不推薦：Option B/C（模糊比對）**

中文字元級 OCR 誤讀 + 子字串重疊的屬性名，模糊比對的 false positive 風險太高，會降低而非提高準確率。

## 9. Domain Knowledge

- 全屬性永遠帶 %，不會有平值
- 爆擊傷害、屬性%、攻擊力% 都是 % 類屬性，平值版本通常不是目標
- 帽子的「技能冷卻時間 -N 秒」是唯一重要的非 % 屬性
- 洗兩排潛能使用絕對附加方塊，相關實作可能需要另外調整
